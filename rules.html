🏭 交貨單稽核系統 - 稽核助手使用說明

本工具自動檢查中機輥輪再生交貨單中的 KEY 錯字、算錯數、漏填資料。

📂 資料提供方式

方式說明
📸 拍照上傳拍攝紙本單據或掃描 PDF，AI 自動讀取文字（約 30-60 秒，超量產生些微 API 費用）。
📊 Excel 上傳直接丟入電子檔，速度最快（1 秒且免費）、準確度最高（表格直讀）。
📂 JSON 上傳用於讀取舊有的系統紀錄檔案。
☁️ 巨集一鍵稽核系統已部署至 GCP Cloud Run，可直接在 Excel 檔案內利用巨集一鍵執行稽核，完全不用離開 Excel 頁面。

1. 📋 表頭資訊稽核 (Header Audit)

檢查項目邏輯與條件
🚨 嚴重混單同一份文件中偵測到多種不同的工令單號或工作申請單編號出現在同一份文件中。
⚠️ 存放位置錯誤依據工令前綴自動判定廠區。例如：W3 開頭必須儲存於 W3 SHOP、W2 開頭為 W2 SHOP、Y461 開頭為 Y461 SHOP、Y463 開頭為 Y463 SHOP、W46 開頭為 W46 SHOP，OW 則須為 中龍 SHOP。
⚠️ 工令格式錯誤單號不符合預設規則（須為 W, R, O, Y 開頭且長度為 10 碼）。內部新增防雜訊機制：O 開頭須含至少 2 個數字，其餘須含至少 6 個數字，且嚴格排除包含 KEY/WAY 等雜訊字眼。
⚠️ 工作申請單編號異常除了檢查混單外，格式必須為 2 開頭，且長度精準限制為 10 碼或 12 碼。
📅 日期不一致與缺失整份文件出現兩個以上的預定交貨日或實際交貨日；或是全卷皆未填寫或無法辨識日期。
⏰ 逾期交貨任一項目的「實際交貨日」晚於「預定交貨日」，系統將自動計算並回報確切延遲天數。
📑 頁碼重複偵測到相同的頁碼數字於全卷中出現兩次以上（防止重複影印或掃描）。
📑 頁碼缺漏/未從頭開始嚴格檢查系統擷取到的實體頁碼是否從第 1 頁開始，且必須連續無中斷。

2. 📏 規格與長約稽核 (Spec & Contract Audit)

檢查項目邏輯與條件
🛑 長約項次比對系統會自動讀取 contract_items.xlsx 執行 100% 嚴格字串比對（經過去除空格、特殊符號後的英數中完美配對）。若完全無法在長約清單中找到吻合項目，即判定為非長約項目並報錯（標題含「報廢」之項目自動豁免此項核對）。
⚠️ 規格取代落差當系統透過 rules.xlsx 強制更換規格後，若「原始辨識內容」與「強制更換規格」相似度低於 100%（系統比對時會自動保留 "MM" 等關鍵單位以確保精準），會觸發異常提醒人工覆核。

3. 🔢 工程與數值稽核 (Numerical Audit)

檢查項目邏輯與條件
⚠️ 規格缺漏/無效偵測到項目名稱，但規格欄位為空；或該項目屬於必須嚴格檢查數值之分類，但規格內容解析不出任何有效數字基準。
🛑 單位異常 (標題括號)標題後方括號內除數字外，單位僅允許填寫「PC」或「SET」，若包含其他文字雜訊將觸發報錯。
🛑 數據格式錯誤/缺漏數值欄位填寫了 N/A、OK、- 或文字雜訊（在嚴格數值分類下）。若明細完全未填寫任何數據，將觸發「數據缺漏」攔截。
🛑 數值異常 (不可為零)全域防呆限制，實測數據絕對不可填寫 0 或 0.0，若無實際數值應填寫 N/A 或是保持空白。
🛑 單位異常 (整批製程)熱處理：數量後方必須標示單位「KG」。
研磨/動平衡：數量後方必須標示單位「IN2」或「IN²」。
*(若標題含「校正、串列式、W3X、內孔、小鋼胚」等白名單關鍵字，則豁免單位檢查)*
🛑 明細異常 (反向檢查)熱處理：僅需填寫總重量，底下的「編號與尺寸」明細欄位必須保持空白，不可填寫。
研磨/動平衡：除了總量與單位，底下的明細必須逐筆填寫。
🛑 報廢異常若標題包含「報廢」，其尺寸明細必須填寫為 N/A 或保留空白，嚴禁填寫實際數值（系統判定為報廢件後，會自動豁免後續常規尺寸邏輯與流程履歷）。
🛑 豁免異常 (空值與純數字)系統強制規定被判定為「豁免」或「SKIP」之項目（不須量測尺寸）：
1. 絕對不可為空：對應欄位絕對不可完全空白，必須有文字註記（如 N/A、OK）。
2. 反向防護：不可填寫「純數字」，避免現場誤填為實際尺寸。
🛑 尺寸異常分類檢驗依不同分類執行專屬防呆演算法：
• 銲補 (Min Limit)：須為純整數，低於下限或超過上限 20 以上即報錯。
• 未再生/粗車：小於目標值須為整數；大於目標值須填寫兩位小數。
• 軸頸上限 (Max Limit)：須為純整數且不可超過最高上限。
• 精加工/研磨 (Range)：必須填寫兩位小數，且精準落在公差區間內。
• 幾何/形位公差：數值不可為負數，且須填寫兩位小數，亦不可超過容許上限。
• 1機310軸頸特規：標題含「1機310」之軸頸，同編號兩端尺寸差值必須大於等於 8mm。
• 驅動輥內孔階梯特規：自動演算法檢查數值集中度（差值 < 1mm），依量測總數限制密集上限（如總數<=4限2個，<=8限4個），防止現場未確實分段量測。
🛑 特規異常 (本體再生車修)針對單一區間且上限小於等於 150mm 的本體再生車修項目，若實測值小於下限，頁面備註必須精確包含「依現場需求調整車修尺寸下限」，否則報錯。
🛑 異常 (多餘備註)全卷反向防呆。若該頁並未觸發上述「車修尺寸小於下限」特規條件（例如數據全在區間內，或為純拆裝頁面），右下角卻填寫了相關尺寸調整備註，系統將主動報錯攔截。
🛑 數據損壞與單格豁免偵測到 [!] 壞軌標記將直接報錯。針對整批項目，若單一數值欄位填寫了數量單位（如 IN, SET, PC, 式），該格自動豁免嚴格數值檢驗，避免誤報。

4. 🧮 會計與數量匯總稽核 (Accounting Audit)

檢查項目邏輯與條件
⚠️ 申請與實交差異總表內「申請數量」不等於「實交數量」，或兩者皆為 0 / 空白。
🛑 單位異常(總表)嚴格核對總表項目的「單位」欄位是否合規：
1. 不可空白：總表該項目的單位欄位絕對不可留空。
2. 熱處理：標題含「熱處理」者，單位必須為「KG」。
3. 面研磨/動平衡：標題含「面研磨」或「動平衡」者，單位必須為「IN2」或「IN²」。
4. 一般項目：其餘常規加工項目，單位僅允許為「PC」或「SET」。
🛑 統計不符(單項)明細標題要求的目標數（如括號內的數量）與實際抄錄有數據之筆數不符（系統會自動結合 rules.xlsx 的單項計價倍率轉換）。
⚠️ 編號重複同一個工件 ID 在本體清單重複出現 2 次以上，或在軸頸、內孔及「鍵槽 (KEYWAY)」清單重複出現 3 次以上（容許上限為 2 次）。
🛑 明細匯總不符總表實交數不等於明細項目經過「運費比重」、「聚合規則 (u_agg)」加總後的總和。
👻 幽靈明細明細項目名稱無法匹配至總表中的任何一項（可能名稱差異過大或真的沒有總表項目）。
🛑 總表缺漏 (Mode B)明細觸發了特定的匯總邏輯（如 ROLL 拆裝、車修、銲補），但總表未宣告該特定項次。
🛡️ 語意防撞過濾系統配對時嚴格防撞，自動隔離「新品」與「舊品」之項目，並在製程、部位、特徵上設有互斥防護，避免會計項目發生錯配。
🌊 運費階層與流道邏輯獨立流道：「新品組裝」直接計入運費。
瀑布流道篩選：舊品拆裝 > 本體未再生 > 本體銲補 > 一般銲補(非軸頸) > 面研磨 > 動平衡。
*(註：目前以實際有效明細件數來計算研磨與動平衡的運費，不以面積計算)*

5. 🔄 流程與溯源稽核 (Process Audit)

檢查項目邏輯與條件
🛑 工序尺寸合理性檢查透過零件歷史履歷進行物理與完工邏輯交叉比對：
1. 不應加工：若「未再生 (Stage 1)」實測值已含小數點（視為完工），後續工序不可再出現任何加工工序紀錄。
2. 尺寸倒置：後續加工工序的實測尺寸與前序尺寸不符物理加工邏輯（例如：粗車尺寸小於精車尺寸）。
3. 銲補與再生差值：本體銲補與再生的實測差值必須介於 4mm ~ 10mm 之間(BCC 350 PPINCH ROLL 5mm ~ 10mm 之間)；軸頸銲補與再生的實測差值必須介於 3mm ~ 7mm 之間，超出範圍即報錯。
🛑 數據缺漏 (流程斷層)關鍵流程項目（本體/軸頸/特殊），若完全未填寫任何有效量測數據（如填寫純 N/A 或無數據），將導致後續溯源斷層。
🛑 流程異常身分衝突1. 組裝/拆裝衝突：同一個零件編號「嚴禁」同時出現在「新品組裝」與「舊品拆裝」明細中。
2. 新品矛盾與豁免：申報為「新品組裝」之零件，嚴禁具備「本體未再生 (粗車)」的加工紀錄。但在履歷溯源時，新品身分會自動豁免「缺少本體未再生前置工序」的斷層報錯。
🛑 流程異常明細/特規1. 重複明細：全卷禁止出現名稱完全相同的重複明細項目（系統將依據去除括號與特殊符號後的名稱進行次數報錯）。
2. Wrapper鍵槽：Wrapper 輥輪兩端鍵槽尺寸差異應大於等於 5mm（通常為 25mm 與 36mm），若差值過小將提醒確認是否填寫或量測錯誤。
🛑 流道異常 (終極防撞)導入終極防撞演算法 (Set Intersection Fingerprinting)：
1. 同層級互斥：同一工序階段中，不同項目嚴禁認領相同的編號（白名單：主動軸與從動軸合法配對除外），防止手滑重複登錄或複製貼上錯誤。
2. 跨工序混單：上下游工序集合必須為完美子集或超集關係。若出現「部分交集錯位」（下游混入上游沒有的越界編號，且上游未完全流向下游），系統將主動抓出跨流道誤填。
🛑 溯源異常 (拆裝池分流)1. 多餘/漏報拆裝：申報拆裝但在未再生清單找不到 ID，或有加工紀錄卻缺失拆裝數據（成人池 1 對 1 檢驗）。
2. 1 拆 2 規則與獨立池：兒童池（如 W3#1~3機、SCC#6機）ID 拆分（-1, -2）數量必須齊全。系統會嚴格區分「一般成人池」與「專屬兒童池」，兩者禁止跨池對應。
🛑 溯源異常 (工序與相依)1. 缺軸頸/缺本體/製程未完：有軸頸加工卻無本體紀錄；有銲補 (Stage 2) 紀錄，卻缺失後續再生 (Stage 3) 紀錄；軸頸項目強制必須包含「未再生、銲補、再生」完整履歷。
2. 軸心/鏟修依賴：出現「軸心」項目時，其編號必須包含在「驅動輥」母體清單內。出現「鏟修」紀錄時，該零件編號必須存在於全卷的「本體未再生」清單中。
🛑 流程異常 (CPLG/DRIVEN)1. 屬性不符：頭尾 (Stage 1/Stage 3) 的 CPLG 屬性不一致，或頭尾皆為 CPLG 但中間銲補工序變為一般屬性。
2. 非法 CPLG：名稱標註 CPLG 但缺乏 VIP 關鍵字（如 #4機270/290/320/350）。
3. 禁用字：CPLG 軸頸規格內禁止出現「非」字。

6. ☁️ 雲端紀錄與例外處理

檢查項目邏輯與條件
✅ 雲端同步交貨單檢查合格紀錄會自動上傳至 Firebase 結構化資料庫，方便後續快速查詢、排序與檢視完整的稽核軌跡。
🛑 開關控制 (STOP_LOGGING)系統支援偵測 STOP_LOGGING 開關（優先讀取 Cloud Run 環境變數，次之讀取 Streamlit Secrets），開啟後將直接阻擋 Firebase 寫入，方便開發測試與除錯。
